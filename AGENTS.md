# mtro_scheduler 프로젝트 규칙

## 필수 규칙

### 아키텍처

- `src/app`은 화면, React 상태, 사용자 상호작용을 담당한다.
- `src/domain`은 프레임워크와 브라우저 API에 의존하지 않는 배정, 검증, 파싱 규칙을 담당한다.
- `src/data`는 CSV와 `localStorage` 등 데이터 입출력을 담당한다.
- `src/export`는 PNG 생성과 파일 저장을 담당한다.
- 새 기능은 위 경계를 유지하고, 도메인 규칙을 React 컴포넌트나 저장소 코드에 중복 구현하지 않는다.

### 테스트와 검증

- 도메인 동작을 변경하거나 버그를 수정할 때는 해당 동작을 검증하는 Vitest 테스트를 추가하거나 수정한다.
- 작업 완료 전 `npm test`와 `npm run build`를 실행한다.
- 테스트나 빌드를 실행하지 못했거나 실패한 경우 그 이유와 영향을 명시한다.

### 브라우저 저장소

- `localStorage` 접근은 읽기, 파싱, 쓰기 예외를 처리해야 하며 저장 실패가 앱 렌더링 전체를 중단시키지 않게 한다.
- 저장 데이터를 타입 단언만으로 신뢰하지 말고 런타임에서 구조를 검증한다.
- 저장 스키마를 변경할 때는 버전을 올리고 기존 데이터의 마이그레이션 또는 안전한 폴백을 제공한다.

### 비동기 OCR

- OCR 요청에는 요청 식별자나 취소 수단을 두고, 최신 요청이 아닌 결과는 상태에 반영하지 않는다.
- 이미지 재선택, 기준월 변경, 초기화, 컴포넌트 해제 시 진행 중인 이전 요청의 결과를 폐기한다.

### UI와 날짜

- 기존 MUI 테마와 컴포넌트 패턴을 우선 재사용한다.
- 날짜와 기준월의 내부 표현 및 표시 형식은 기존 프로젝트 관례를 유지한다.
- 화면 이동을 변경할 때는 현재 hash 기반 경로와 브라우저 뒤로가기 동작을 보존한다.

### 개인정보와 내보내기

- 명단 CSV와 OCR 입력에 포함된 개인정보를 외부 서비스로 전송하거나 로그에 노출하지 않는다.
- CSV 및 파일 내보내기 값은 수식 주입과 구분자, 따옴표, 줄바꿈을 고려해 안전하게 escape한다.

### 배포와 PWA

- 정적 자산과 라우팅은 `/mtro_scheduler/` 배포 기준 경로에서 동작해야 한다.
- 서비스 워커와 PWA 설정을 변경할 때는 기존 업데이트 흐름과 오프라인 동작을 확인한다.

## 권장 규칙

- `App.tsx`에 내비게이션, 검증, 상태 연결을 계속 집중시키지 말고 책임이 커지면 화면 또는 훅으로 분리한다.
- `tesseract.js`처럼 무거운 선택 기능의 의존성은 필요 시점에 동적 import하여 초기 번들에 포함되는 범위를 줄인다.
- 저장소, OCR 경합, 명단 편집, 뒤로가기, 내보내기 등 브라우저 경계 동작에는 통합 테스트를 우선 보강한다.
- 설정, 명령어, 공개 API 또는 문서화된 동작이 바뀌면 관련 문서를 함께 갱신한다.
